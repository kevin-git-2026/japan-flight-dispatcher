# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

A Python desktop GUI tool that generates Japan-region flight assignments for X-Plane 11/12 flight simulator pilots. It picks departure/arrival airports (random or fixed), looks up recommended AIP routes between them, and cross-references real-world flight schedules. An optional Volanta integration (F11) reads the user's flown-route history to bias random planning toward un-flown routes. The shipped artifact is `flight_dispatcher.exe` (a PyInstaller `--frozen` build). User-facing strings and `README.md` are in Simplified Chinese; keep that convention when editing output.

As of v1.2.0, the code was split out of the original single 1100-line `flight_dispatcher.py` into a small package: `flight_dispatcher.py` is now a **thin entry shim**, and the logic lives in the `dispatcher/` package (one module per concern — see Architecture). As of **v1.3.0** the front-end is a **tkinter GUI** (`dispatcher/gui.py` → `run_gui()`); **as of v1.3.1 the GUI is the only front-end** — the original interactive CLI (`dispatcher/app.py`) and its `--cli` flag were removed. The entry shim now just does `from dispatcher.gui import run_gui; run_gui()`. Mostly stdlib; **as of v1.4.0 the interactive map (F18) adds third-party deps `tkintermapview` + `Pillow`** — imported defensively (`_HAS_MAP`), so if they're missing the map link just hides and everything else still runs stdlib-only.

## Commands

There is no build system or test suite. Historically stdlib-only; **as of v1.4.0 the map feature (F18) pulls in `tkintermapview` + `Pillow`** (`pip install tkintermapview`) — for the frozen build add `--collect-all tkintermapview` to PyInstaller, and ship `aircrafts.json` next to the exe. Everything except the map degrades gracefully if those libs are absent.

- Run from source: `python flight_dispatcher.py` → tkinter window (the only front-end since v1.3.1; there is no non-interactive/headless mode).
- Build the distributed GUI exe: `pyinstaller --onefile --windowed flight_dispatcher.py` (`--windowed` hides the console; frozen mode detected via `sys.frozen`). The GUI redirects `sys.stdout`/`stderr` into its log pane — which is also why `--windowed` (where stdout is `None`) is safe for all the reused functions that `print()`.

**Import-path note:** `dispatcher/config.py:get_real_run_path()` anchors every sibling file (`NavData/`, `installed_scenery.json`, `routes_cache.csv`, `volanta_data.json`). In source mode it returns the **parent of the package dir** (`dirname(dirname(__file__))` = project root), NOT the module's own dir — so the package layout resolves the same root the old single file did. Frozen mode still uses `sys.executable`'s dir. Don't "simplify" this back to a single `dirname`.

Data flows: locate navdata (bundled `NavData/` only) + AIRAC check → scan installed scenery (XP + MSFS) → load AIP routes → (optionally sync & load Volanta flown routes) → each **规划航线** click weight-picks/matches airports by distance, finds AIP routes, and scrapes real flights.

**Navdata discovery** (`find_navdata_file`): finds `earth_aptmeta.dat` in the **bundled `NavData/` folder** next to the program (primary and only source; fully XP-decoupled, so MSFS-only users without X-Plane work too). If it's absent the GUI shows a messagebox pointing to Navigraph (`navigraph.com/downloads`, "X-Plane 12"). The old CLI had extra fallbacks — exe-sibling `.dat`, X-Plane install auto-locate, manual paste — bundled in `find_xp_data_files`, which was removed with the CLI in v1.3.1 (`locate_xp_root` itself stays, used by the scenery scanner). `scenery_packs.ini` is now **optional** — scenery detection moved to the scanner below. `check_airac_currency` reads `NavData/cycle_info.txt` and warns (with month-abbr parsing that doesn't depend on locale) if the AIRAC cycle has expired. `get_real_run_path()` anchors all sibling files (`installed_scenery.json`, `routes_cache.csv`, `volanta_data.json`).

**Airport loading** (`load_airports_from_navigraph`): parses the whitespace-delimited `.dat`, keeping only ICAO regions `RJ`/`RO` (Japan) with runways ≥ `min_runway_ft`. Marks `is_military` from a type flag and sets `scenery_sources` (a set like `{'XP','MSFS'}`, or `None` if scenery wasn't detected) from the scanner's `scenery_map`; `has_scenery` is a derived property. Each row becomes an `Airport` object (code + decimal lat/lon).

**Multi-source scenery scan** (`scan_installed_sceneries` + helpers, F5): detects installed airport scenery across **both sims**, returning `dict{ICAO: set('XP'|'MSFS')}`. XP: `locate_xp_root` → `Custom Scenery`, each pack's `Earth nav data/apt.dat` parsed for airport-row ICAOs. MSFS: `find_msfs_packages_dir` reads `InstalledPackagesPath` from candidate `UserCfg.opt` paths (covers 2020/2024 × Steam/MS Store, with a drive-scan fallback) → `Community`; per pack `_extract_msfs_pack_icaos` runs a 4-step cascade (folder name → `ContentInfo/…/ContentHistory.json` `type=="Airport"` content → `scenery/*.bgl` filenames → give up), pre-filtered by `manifest.json content_type=="SCENERY"`. Only `RJ/RO` ICAOs are kept, then **whitelisted against the real-airport ICAO set from navdata** (`load_japan_icao_set` parses every `RJ/RO` code from `earth_aptmeta.dat`): this drops false positives where the `R[JO]xx` extractor matched an ordinary word inside a folder/bgl name (e.g. `ngt-road-mesh`→`ROAD`, `aerocaches`→`ROCA`) — a real airport set is the only thing that distinguishes `ROAD` (not an airport) from `ROAH` (Naha). The extractor regex (`_extract_japan_icaos`) is also word-boundary-anchored as a first line of defense; the navdata whitelist is the authoritative one (skipped if navdata is missing, to avoid wiping everything). Results + a lightweight fingerprint (`{scenery_dir: {pack: mtime}}`) cache to `installed_scenery.json`, so unchanged dirs return instantly (a stale cache that still holds pre-fix false positives is self-healed/rewritten on the next cache-hit read); a changed pack-set/mtime triggers a rescan. The same JSON also holds `xp_root`/`msfs_packages` (migrated from the legacy `xp_path_config.txt`). MSFS recall isn't 100% (manifest has no ICAO field; oddly-named packs without `ContentHistory.json` are missed) — it's a soft "reference" hint.

**AIP route data** (`load_aip_routes_from_csv`): network-first — downloads `https://jp-routes.vercel.app/public/routes.csv` (5s timeout), refreshing the local `routes_cache.csv` mirror. Falls back to the cache offline (warns past a 28-day staleness window). The CSV columns are `DEP,DEST,Time Restriction,Altitude,Aircraft,Route,Remarks`; `find_aip_route` filters rows by exact dep/dest ICAO match.

**Real-flight scraping** (`fetch_real_flights_with_filter`): HTTP-GETs a FlightAware `findflight` page and regex-extracts an embedded `FA.findflight.resultsContent` JSON array. Filters by airline ICAO prefix, aircraft type (fuzzy via `is_aircraft_match`), and departure time window. Returns up to 5 matches; if none match the filters, returns up to 5 unfiltered alternatives; if the scrape returns **zero** flights, the caller falls back to a sim callsign whose airline is chosen by `pick_sim_airline(dep, arr)` (`dispatcher/airlines.py`) — a route-aware pick that only allows airlines whose network covers both endpoints' geographic regions, so a Hokkaido-based carrier (ADO) won't be assigned a Fukuoka–Okinawa route (a user-specified airline still wins). The airline→region networks and airport→region map live in an editable `airlines.json` (run-dir, auto-created from `_DEFAULT_DATA` on first run; corrupt/missing → built-in defaults), keyed by airline ICAO so users can add/remove/edit carriers (new airlines, VATSIM virtual airlines) without touching code. This depends on FlightAware's page structure — the regex (`FA.findflight.resultsContent = [...]`) is the fragile coupling point.

**Volanta flown-route reading** (`load_volanta_flown_routes` + helpers): gets the user's Volanta (`fly.volanta.app`) flight history to bias random planning toward un-flown routes (F11). The data comes from the **`/api/v1/Flights` API**, fetched directly with the user's own browser session token (see next paragraph); the parsed directed-route counts are stored in the single **`volanta_data.json`** (run dir), which also holds the sync preference and last-fetch time (it consolidates the old `volanta_config.txt`/`volanta_flights.json`/`volanta_flown.json` — first run migrates+deletes them). `load_volanta_flown_routes` reads that store, folds in any user-dropped `volanta_flights.json` (DevTools export — absorbed then deleted) and `volanta_flights.csv` (kept), and returns `dict{(dep,arr): count}`; pure stdlib, every step wrapped in try/except → empty dict on failure. `sum(counts.values())` = total flights, `len(dict)` = distinct directed routes; self-loops (`dep==arr`) are skipped. (Historical note: an earlier version regex-scanned the IndexedDB LevelDB for routes — `find_volanta_leveldb_dirs`/`extract_flown_routes` — but that was fragile (V8 dedup) and incomplete (lazy-load), and has been **removed** in favor of the API path. `_read_leveldb_text` survives, now used only to read the Local Storage LevelDB for the session token.)

**How the data is fetched — `/api/v1/Flights` via the browser session** (`try_fetch_volanta_json_via_session`): the **`/api/v1/Flights` API** returns ALL flights in one clean JSON call. The program calls it directly using the user's **own** Volanta session token read from their **own** browser — analogous to yt-dlp's `--cookies-from-browser`. Key details discovered by reverse-engineering:
- The API rejects the Firebase idToken (`iss: securetoken.google.com/volanta` → `error="invalid_token", The issuer ... is invalid`). The token it accepts is an **Orbx-issued JWT** (`iss: "Orbx"`, `aud: "regular_user"`, valid **~14 days**) — Volanta is an Orbx product. That token lives in **Local Storage** (`<Profile>/Local Storage/leveldb`), NOT IndexedDB.
- `_extract_volanta_api_token` scans the Local Storage leveldb but **only** accepts JWTs with `iss=="Orbx"` AND a `fly.volanta.app` marker within 3000 chars before them — so it never extracts/uses any other site's token from the shared Local Storage. `_fetch_volanta_flights_json` GETs the API with `Authorization: Bearer <token>` and `Accept-Encoding: gzip` (stdlib has no brotli). The token is used in-memory for that one request only — **never written to disk, never sent anywhere except `api.volanta.app`**.
- `try_fetch_volanta_json_via_session` parses the API response and merges the authoritative counts into `volanta_data.json` (atomic temp→replace, only if it parses to ≥1 route; the raw ~900KB response is NOT kept on disk). `skip_if_fresh=N` skips the network call if the store's `fetched_at` is younger than N seconds (startup uses 3600).

The GUI only auto-fetches **after the user has opted in** (see the preference model below). `_load_volanta_json` parses the file (each item is `{flight:{...}, summarisedPositions:[...]}`; route = `flight.originIcao`/`flight.destinationIcao`). When opted-in the silent fetch needs **zero** steps (no DevTools, no scrolling) as long as the user logged into Volanta in a browser within ~14 days; if the token expired it degrades to the existing accumulated store. Either way `load_volanta_flown_routes()` reads any present json/CSV/store.

**Security boundary (important for maintainers)**: this reads the user's own session token from their own browser, locally, for their own data — no third party, no exfiltration. The earlier in-session classifier block was on *exploratory* token-scanning; this is a *targeted, user-authorized* feature scoped to one app. Keep it that way: only `iss=="Orbx"` + `fly.volanta.app`-proximity tokens, only for `api.volanta.app`, never persisted/logged/transmitted.

**Persistent store + multi-source merge**: the **monotonic** flown-route store lives in `volanta_data.json`'s `flown` field (run dir) — `{"DEP|ARR": count}`, **only grows**. Merge rule: **route keys are unioned** (a flown route is never lost); **counts prefer the authoritative source** (the API fetch, or a user-dropped `volanta_flights.json`), falling back to `max()` across CSV/store for routes it lacks. The merged result is persisted back via an atomic write. For weighting only "flown (≥1) vs un-flown (0)" matters, so exact counts are non-critical — but authoritative counts keep the displayed total honest. `meta['latest']` is the store's `fetched_at` (when the API was last pulled).

**Sync preference model** (`volanta_data.json`'s `preference` field, run dir): `auto` = opted-in, anything else (missing / `ask`) = not-opted-in. The **GUI** drives it via `volanta_auto_enabled()` / `set_volanta_auto()`:
- **opted-in (`auto`)** → the init worker silently `try_fetch_volanta_json_via_session(skip_if_fresh=3600)` on startup (no prompt; **only here do we ever scan the browser's Local Storage**), then `load_volanta_flown_routes()`. Token-expired → degrades to the store. The **自动同步** checkbox shows checked.
- **not opted-in** → no startup scan; the user clicks **同步 Volanta** to sync on demand (see Browser sync below). Checking **自动同步** calls `set_volanta_auto(True)` → future startups auto-sync; unchecking writes `ask`. Either way `load_volanta_flown_routes()` still loads any manually-placed json/CSV/store.

**Browser sync** (GUI `_volanta_worker`, 5-min cap): when **同步 Volanta** is clicked and no valid token is present, `_open_volanta_in_browser()` opens **`fly.volanta.app/map`** (`_VOLANTA_LOGIN_URL`) in Edge (falls back to default browser) so the user can **log in**. A daemon thread polls `try_fetch_volanta_json_via_session()` every 3s (cancelable via `threading.Event`, `_VOLANTA_POLL_CAP`=300s) until the Orbx token is readable on disk. **Root cause of the alpha1 sync timeout (fixed 2026-06-28)**: the Orbx token IS minted right after login on `/map` (no `/flights` needed — an earlier "/map doesn't mint it" hypothesis was wrong), but Chromium flushes the renderer's in-memory localStorage to the on-disk leveldb (which we read) with a ~30s–1min lag (longer when idle, sometimes >180s) — so the old 180s poll sometimes timed out; scrolling/navigating generates localStorage writes that force an earlier flush (why the user's manual `/flights`+scroll worked). Fix: 300s cap + two `messagebox` popups (①"waiting for write" at start, ②at ~60s, guide the user to open `/flights` and refresh+scroll to force the flush) + a success popup — the program does NOT auto-open `/flights` (avoids the pre-login `/flights` hang). Plus `volanta.diagnose_volanta_session()` → privacy-safe category (`token_ok/no_dirs/empty/orbx_expired/no_orbx`, never any token value) and `try_fetch_volanta_json_via_session(diag=)` logging one fixed reason line on failure (`diag` is on only for the fast-path + startup auto-sync, NOT the 3s poll, to avoid log spam). The `_extract_volanta_api_token` proximity heuristic was verified fine and left unchanged.

**Route selection** (`get_random_route`): enumerates all airport pairs whose great-circle distance (`calculate_distance_nm`, Haversine in NM) falls in `[min_dist, max_dist]` (and, with `strict_aip=True`, exist in a prebuilt `aip_index` set; and, with `require_both_scenery=True` — the random-only GUI toggle — both endpoints satisfy `Airport.has_scenery`, which is inert when scenery wasn't detected since `has_scenery` is then `True`). Candidates are bucketed into 3 tiers by **military-endpoint count** (0 = both civil, 1 = one military, 2 = both), and selection happens within the **lowest non-empty tier** — so civil-only routes are strictly preferred and military airports (which often lack civil nav aids / approaches, so the route may be unflyable) are only picked when no civil route fits the constraints or when the user fixes a military endpoint. This matters for heavy flyers: once almost every civil route is flown, the high un-flown weight would otherwise spike military picks; the tiering prevents that (it re-flies the least-flown civil route instead). Within the chosen tier it weighted-random-picks via `random.choices` with weight `1/(count+1)²`, where `count` is the Volanta flown count — soft-prefers un-flown routes (empty `flown_counts` → uniform, matching the original behavior). Returns a 5-tuple `(dep, arr, dist, route, flown_count)`. (This replaced the older 150k-iteration rejection-sampling loop.)

**Startup + planning flow** (`dispatcher/gui.py`, invoked by the `flight_dispatcher.py` entry shim → `run_gui()`): a background `_init_worker` does the one-time setup (navdata discovery + AIRAC check + scenery scan → `scenery_map` + airline data + AIP load + preference-gated Volanta sync + prebuild `aip_index`), posting results back to the Tk thread via `after()`. Each **规划航线** click runs `_plan_worker` on a daemon thread: airports are *reloaded* every time via `load_airports_from_navigraph` (passing `scenery_map`) because `min_runway_ft` is a per-plan input, then route selection → `planner.build_flight_plan` → `FlightPlan`. `_render_plan` renders it into the result widget — scenery labels via `Airport.scenery_label()` (`[地景:XP/MSFS/XP+MSFS]` or `[⚠️无地景]`), plus a `已飞过 N 次` note when `flown_count > 0`. The compute (FlightAware scrape + sim-callsign) lives in `planner.build_flight_plan`, decoupled from rendering so a future GUI-framework swap only touches `gui.py`.

**Local A\* route generation** (`dispatcher/router.py`, F15, **v1.4.x in progress**): when no direct AIP route exists, generate an enroute airway route from the bundled X-Plane navdata. Priority chain: **Rule 0** (direct AIP) is the app-level gate (`gui._plan_worker` calls `find_aip_route` first; `generate_route` is only invoked when it returns None). Inside `generate_route(dep, arr, dat_path, aip_data, airports)` (`airports` now unused, kept for back-compat): it builds a lazy/cached `AirwayGraph` (parse `earth_fix`/`earth_nav`/`earth_awy`, Japan bbox `(24,46,122,149)`, ~2406 nodes/4582 edges, key `(ident,region)`, honoring airway direction N/F/B), then `_direct_route` runs **A\* between endpoints** chosen by `_departure_candidates`/`_arrival_candidates`: the **union** of (i) **learned dep/arr transition fixes** from `routes.csv` (`graph.dep_heads`/`arr_tails`), (ii) **VATJPN transfer-point gates**, and (iii) **CIFP** procedures — departure = SID exits (section `E`) ∪ **on-field VOR** (`_onfield_vor`: nearest section-D VOR within 15nm — NOT any section-D VOR, which would grab a mid-SID/feeder VOR like RJSY's YTE at 37nm), arrival = STAR entries → on-field VOR → IAF/IF — all `_dir_filter`'d (direction) + connectivity-filtered (on the airway network), geometric nearest-toward as last fallback; A\* self-selects the best start/exit. A continuity check flags enroute acute turns (`_MAX_TURN_DEG`=100°) → `FlightPlan.generated_route_warn`. Result is rendered as a distinct "🧭 生成航路 (非官方 AIP)" section. *(The old **Rule 5 AIP bridge** `_try_aip_bridge` was removed in v1.4.1_alpha2 — see 续作3 below.)* Domain rules in the `route_planning` skill (`.claude/skills/route_planning/`); full design in `revisions.md` v1.4.0/续作3.

**Route quality (续作2)** — layers that make generated routes match real-world operation, all learned from `routes.csv` (official published routes never fly an airway illegally): (1) **RNAV preference** — A\* penalises pure-traditional edges (`_TRAD_AIRWAY_PENALTY`=1.15) and `_format_route`/`_pick_airway` prefer RNAV names (Q/T/Y/Z·L/M/N/P); RNAV share 74→92%. (2) **Airway-direction learning** — `earth_awy` marks a one-way RNAV airway `N` where it shares a physical segment with a two-way airway, losing its direction; `_learn_routes` walks every `routes.csv` route → `legal_seg{(u,v):airways}` so `_pick_airway` only labels a one-way RNAV in its attested direction (fixes Y284/Y43/Y312 reversals); label-layer only, graph topology unchanged. (3) **Corridor weighting** — same learning yields `seg_pop` (per-directed-segment popularity); A\* prefers trunk corridors (`_TRUNK_POP`/`_MINOR_CORRIDOR_PENALTY`/`_OFFTRUNK_PENALTY`), all multipliers ≥1 (admissible); trunk share 73→80%, +0.4% distance. (4) **Transfer-point endpoints** — `transfer_points.json` (VATJPN「移管点と高度」SOP, 56 airports' arrival/departure gates) loaded by `_transfer_points()`, `_dir_filter`'d (drop off-direction gates, `_GATE_DIR_MARGIN_NM`=30) + departure quality give-up (`_GATE_GIVEUP_RATIO`=1.4: if the gate route is suspect or ≫GC, drop it → on-field-VOR/geometric, i.e. "本场台离场+VOR 程序").

**AIP-bridge × corridor fusion (续作3, v1.4.1_alpha2)** — the insight that endpoint-learning subsumes the old AIP bridge: (1) **Endpoint learning** — `_learn_routes` (was `_learn_airway_directions`) in the SAME `routes.csv` pass also extracts each airport's **real transition fixes** → `graph.dep_heads`/`arr_tails` (e.g. RJGG-north departs KCC, RJFM-east departs MADOG); only when the route string's first/last token is itself a fix (airway-leading rows like `Y14 HWE …` would otherwise pick a far-downstream fix → degenerate "DCT 200nm to a midpoint", e.g. RJCN→RJSM collapsing to `HWE`). `_learned_heads`/`_learned_tails` read them. (2) **Union endpoint selection** — `_departure_candidates`/`_arrival_candidates` union learned ∪ transfer-gate ∪ CIFP (dir-filtered), A\* self-selects (monotonically ≥ any subset, so strictly non-worse than CIFP-only). (3) **On-field VOR fix** — `_onfield_vor` = nearest section-D VOR within `_ONFIELD_VOR_NM`=15nm; the old code treated *every* section-D VOR as the on-field VOR, wrongly picking mid-SID waypoints (RJSY's YTE, a ZUNDA2 waypoint 37nm out) — this matches the `route_planning` skill's own `cifp_format.md` rule. (4) **Rule-5 AIP bridge removed** — `_try_aip_bridge` + constants `_BRIDGE_TOLERANCE`/`_OVERSHOOT_NM`/`_BRIDGE_HEAD_NM` deleted; with correct learned endpoints the direct A\* is theoretically optimal, and the bridge's borrowing of a *neighbor airport's* departure fix caused backtracks (RJFM→RJBE wrongly via Kagoshima's MIDAI). 50-sample after removal: total distance dropped 22832→22682nm, trunk 78%, 0 degenerate routes. All multipliers/penalties are tunable module constants. `transfer_points.json` is run-dir-anchored (ship next to the exe in frozen builds, like `airlines.json`/`aircrafts.json`). `DEBUG_CORRIDOR=True` logs each route's corridor breakdown. *(The skill's `SKILL.md`/`enroute.md` still describe a "case 5" bridge — a valid domain concept, but no longer this project's implementation; pending cleanup.)*

## Conventions & gotchas

- All console output and prompts are Chinese with emoji prefixes; match the existing tone/format when adding output.
- External dependencies are two live web services (`jp-routes.vercel.app`, `flightaware.com`) — both are scraped, not official APIs, so both can break without notice. There is no caching for FlightAware results.
- `min_runway_ft` accepts user units (`1800m` or `5900ft`) and normalizes to feet; default 5900 ft.
- `installed_scenery.json` (run dir) unifies sim-dir memory (`xp_root`/`msfs_packages`) + scenery cache (`sceneries` + `fingerprint`); it supersedes the legacy `xp_path_config.txt` (migrated on first run). These config/cache files are environment-specific and overwritten at runtime. `airlines.json` (run dir) is the editable airline/region data for `pick_sim_airline` — auto-created from defaults on first run, but unlike the cache files it is meant to be hand-edited and is only regenerated if absent (a corrupt one falls back to built-in defaults without being overwritten). `transfer_points.json` (run dir, v1.4.0 续作2) holds per-airport arrival/departure gates parsed from the VATJPN transfer-of-control SOP (`route_planning` skill `transfer_points.md`); shipped static data the router reads for endpoint selection (missing → falls back to CIFP endpoints). **`arr_dct` (v1.4.1)**: per-gate post-gate **DCT arrival tails** (`{gate: [fixes]}`) — the off-airway direct fixes the VATJPN「到着」routing flies after the gate to the field, which the router appends to the generated route (see router.py row).
- Scenery detection reads the sims' on-disk scenery folders (XP `Custom Scenery` apt.dat, MSFS `Community` packages), not an API — fragile to MSFS pack-naming conventions; any failure degrades to "scenery unknown" (`scenery_sources=None`, no `[⚠️无地景]` mark). Navdata ships in the bundled `NavData/` folder; `check_airac_currency` nudges the user to refresh it via Navigraph (`https://navigraph.com/downloads`, "X-Plane 12" navdata) when the AIRAC cycle expires, and the missing-navdata error in `load_airports_from_navigraph` points to the same link.
- Volanta integration reaches the user's data via Volanta's own `/api/v1/Flights`, authenticated by the Orbx session token read from the **browser's Local Storage** (Edge/Chrome/Brave only — Chromium LevelDB format; **Firefox/Safari use a different store and aren't supported yet**, a cross-platform-round item, and the path logic is Windows-centric) — not the desktop app, not IndexedDB (that old regex path is gone). Any failure degrades to disabled. All Volanta persistence is consolidated into the single **`volanta_data.json`** (run dir): `preference` (`auto` = opted-in → silent auto-sync on startup; missing/`ask` = no startup sync, use the 同步 Volanta button), `fetched_at` (last API pull), and the **monotonic** `flown` store (`{"DEP|ARR": count}`, only grows). Deleting the file fully resets both preference and flown history (there's no separate file to wipe). First run migrates+deletes the legacy `volanta_config.txt`/`volanta_flights.json`/`volanta_flown.json`. Optional user-dropped imports: `volanta_flights.json` (DevTools export — absorbed then deleted) and `volanta_flights.csv` (kept). The browser is only ever opened after the user clicks **同步 Volanta** (or has opted into auto) **and** only when no valid token is already present.
