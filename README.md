# ✈️ 日本航班智能搜索与规划 · Japan Flight Dispatcher

> 为 X-Plane 11/12（兼容 MSFS）的飞行模拟玩家自动生成「日本地区」飞行任务：智能抽取或指定起降机场 → 查询机场间的官方 AIP 推荐航路 → 交叉比对现实世界航班排班 → 输出一份可直接用于模拟飞行的航班规划单。

**当前版本：v2.0.0_alpha1** ｜ 平台：Windows（兼容类 Unix）｜ 形态：纯图形界面桌面程序（`flight_dispatcher.exe`，单文件约 54 MB，冷启动约 2 秒）

## 📥 下载与安装

1. 前往 **[Releases](https://github.com/kevin-git-2026/japan-flight-dispatcher/releases)** 下载最新版压缩包。
2. 解压后，请保持以下文件 / 文件夹在**同一目录**：
   - `flight_dispatcher.exe`
   - `NavData/`（导航数据，详见下方说明）
   - `aircrafts.json`（机型库）
   - `transfer_points.json`（进离场移管点）
   - `operation.json`（机场运行规则，**已附带一套完整的羽田 RJTT 规则作样例**；可用「⚙️ 编辑机场运行规则」照着改写其它机场，删掉该文件即为空规则）
3. 双击 `flight_dispatcher.exe` 即可打开窗口。

> 💡 程序首次运行会在同目录自动生成 `installed_scenery.json`、`airlines.json`、`routes_cache.csv` 等缓存 / 配置文件，属正常现象。

## 🚀 使用

打开后：左侧填写规划条件（**留空＝随机 / 默认**），点 **「🛫 规划航线」**，结果显示在右侧航线卡，底部是运行日志；Volanta 同步、「仅在有地景的机场间规划」等都在界面上点选。

| 输入项 | 示例 | 留空行为 |
| --- | --- | --- |
| 出发 ICAO | `RJAA` | 随机 |
| 目的 ICAO | `RJBB` | 随机 |
| 执飞航司 | `ANA` | 不限 |
| 机型 | `737`（可搜索下拉） | 跳过 |
| 时间区间 | `08:00-15:30` | 跳过 |
| 最短跑道 | `1800m` 或 `5900ft` | 5900 ft |
| 航程范围 | `200` — `450`（NM） | 200 / 450 |

## ⚠️ 使用前请注意

### 1. 导航数据（NavData）

- 程序根目录自带一个 `NavData` 文件夹，直接读取里面的导航数据，**不再强制依赖 X-Plane 安装目录**——即使您只飞 MSFS、根本没装 X-Plane 也能用。
- 请保证 `NavData` 文件夹与 `flight_dispatcher.exe` 放在同一目录下。
- 导航数据有有效期（AIRAC 周期，约 28 天）。每次启动会自检是否过期，过期会提示您前往 **[Navigraph 下载页](https://navigraph.com/downloads)** 重新下载「X-Plane 12」的导航数据，用新数据替换 `NavData` 文件夹后重启即可。

### 2. 联网

- 查询 AIP 航路（jp-routes）、现实航班排班（FlightAware）、加载交互地图瓦片均需联网。
- 首次联网后会生成 AIP 航路缓存，之后可离线查询 AIP 航路；但查询现实航班始终需要联网。

### 3. 地景检测

- 自动扫描 X-Plane 的 `Custom Scenery` 与 MSFS 的 `Community` 文件夹，标注机场地景来源（`[地景:XP]` / `[地景:MSFS]` / `[地景:XP+MSFS]`，未装则标 `[⚠️无地景]`）。
- 结果缓存到 `installed_scenery.json`，目录未变则下次启动秒开；新增 / 删除 / 替换地景包会自动重扫。
- 没找到任何模拟器地景目录时自动跳过（不标注地景），不影响正常规划。
- 随机规划时可勾选「仅在两端都已安装地景的机场之间规划」，配合 Volanta 优先未飞，专挑您装了地景、又还没飞过的航线。

### 4. Volanta 优先未飞航线（可选）

- 如果您用 Volanta 记录飞行，程序会读取您已飞过的航线，在随机规划时**软优先**您还没飞过的航线（不会完全排除已飞航线）。
- 不使用 Volanta 的用户无需任何操作，不影响其它功能。
- 同步方式：程序用您**浏览器里自己的 Volanta 登录会话**自动获取您的全部航班数据（类比 yt-dlp 的「从浏览器读取登录态」）。若本机无有效登录，会打开 `fly.volanta.app/map` 让您登录一下（登录凭据约 14 天有效）。
- **仅支持 Chromium 系浏览器**（Edge / Chrome / Brave）；Firefox / Safari 暂不支持自动获取（跨平台支持是后续重点）。
- **隐私**：登录令牌只用于这一次请求，**绝不保存、绝不发送到任何第三方**。
- 同步偏好、已飞航线累积库、最后获取时间统一保存在 `volanta_data.json`。

### 5. 机型库 / 交互地图 / SimBrief（v1.4.0）

- 机型库 `aircrafts.json` 请与 `flight_dispatcher.exe` 放在同一目录（已随程序分发）；机型输入框可直接搜索 / 下拉选择。
- 航路交互地图：结果卡每条航路有「🗺️」链接，点开在真实地图上画出该航路，可拖拽缩放。一次规划若有多条航路（如分时段的不同 AIP 航路），会在同一张地图里做成**标签页**，切换比对。
- SimBrief 一键派遣：点结果卡上的「🛩️ SimBrief 一键派遣」链接，用您自己浏览器里已登录的 SimBrief 出专业飞行计划（OFP）——无需填写任何账号凭据。

## 📋 更新日志

### v2.0.0_alpha1 → v2.0.0_alpha2
> 这一版全是**航路生成与进离场衔接**的修复，由实飞测试挖出来。追根究底，它们是**同一个病的几种表现**：我们把日本管制移交表（VATJPN）里的信息**只用了三分之一**。

- 🧭 **进场程序（STAR）不再选错**。官方到着表里写的 `…AGPUK MIRAI ABENO IKOMA`，后面那串航点**就是 STAR 本身被逐点展开的样子**（这里是 `IKOMAE`）。程序原先把它们当成普通航路点硬拼进航路，结果航路「吃掉」了 STAR 的入口，进离场面板反而匹配不上任何 STAR，只好把该机场**所有** STAR 列出来、按字母序默认选第一个 —— 于是从关西飞百里（RJAH）时，默认给了往北绕出去再折回来的 `GOT1`，而正解是 `TATSU`。
  - 现在程序会拿这串展开的航点**反查出是哪条 STAR**，直接按官方配对预选。同一个门配多条 STAR 时（如宫崎 KUE 门配 `MELAR`/`KARAH`），再按跑道分：09 号跑道选 `MELAR`、27 号选 `KARAH`。
  - 这套反查同时区分了**两种「直飞」**：能反查到 STAR 的，是被展开的程序（交给 STAR 飞）；反查不到的，才是真正的无程序直飞径路（如熊本 `KUE ESLIL HIGOH KGE`），照旧写进航路。
- 🛬 **进场门按「来向」选**。官方表里每个门前面都写着它服务的**上游走廊**（如中部 `SWING` 来自 `MBE Y121`、`KCC` 来自 `MIDER Y28`）——门本身的位置并不说明它服务哪个方向。补上这层后，从西边飞成田只会考虑 `RUTAS` 门，不会误选北边的 `SWAMP`。
- 🔧 **成田等机场的进场门原先根本没被学到**。程序判断「这个点在不在航路网上」时，两端用了同一个标准；但**进场门常常是航路的终点**（如成田 `RUTAS` 只进不出），于是被整个跳过、退到上游一个**根本不是门**的过境点。修好后，全库 **79 个机场**的官方进场点**无一遗漏**。
- ✈️ **不再飞到本场导航台上空再兜回来**。名古屋小牧的 `KCC` 导航台就在机场里（0.8 海里），程序算距离时它「零成本」，于是不管从哪个方向飞来都选它——可小牧的 5 条 STAR **没有一条**是从 `KCC` 开始的。现在程序会优先选一个**接得上 STAR** 的进场点（绕路在 20% 以内才换）。东京→小牧因此走回了现实中的内陆航路 `GUSRO Y20 SUGAL Y88 SWING`。
- 🔄 **航路不再原地掉头**。A\* 寻路原先对「掉头」不收费，导致会出现接近 180° 的发夹弯（如 RJOA→RJTH 的 100° 急转）。现在超过 30° 的转弯按角度折算成额外里程。
- 📏 **超短程航段改走直飞（会明确告知）**。对马→大分只有 45 海里，程序原先却按航路网算出 **174 海里**（先飞离目的地、再折回来）。这类航段现实中官方本就走**直飞**。现在程序会自动降级为直飞，并在结果卡上**明确标注「已降级为直飞(DCT)，请自行核对」**。
- 🐛 修掉一个显示 bug：规划到**没有 STAR** 的机场（如福江 RJFE）时，STAR 输入框里会残留上一条航线的程序名。

> 效果（200 条随机航路回归）：**大角度掉头的可疑航路 9 条 → 4 条**，**总里程反而少了 665 海里**，进场程序能正确衔接的从 54 条升到 84 条。剩下 4 条可疑航路都是**没有公布进场程序的军用/小型机场**，程序会在结果卡上警告，请自行核对。

## ⚙️ 技术说明

- **核心逻辑全部为 Python 标准库实现**（航路生成、天气、运行规则、Volanta、抓取等，无任何第三方依赖）；**界面层**用 [Flet](https://flet.dev)（Python 编写、Flutter 渲染，v2.0.0 起）。v1.x 的 `tkintermapview` + `Pillow` 已随 tkinter 一并移除。
- 从源码运行：`pip install flet flet-map` 后 `python flight_dispatcher.py`（Python ≥ 3.10）。
- 外部数据源：航路（jp-routes）与排班（FlightAware）为页面抓取而非官方 API，可能随对方改版而失效；天气为 NOAA `tgftp`（METAR/TAF）与 **[Open-Meteo](https://open-meteo.com)** 网格模型（缺测 / 过期时的回退）；**羽田实测运用状况来自国土交通省 [ntrack](https://www.ntrack.mlit.go.jp/NtrackTop/show)**（全公开数据，取不到时自动回退到运行规则）。
- 全部用户界面文案为简体中文。
